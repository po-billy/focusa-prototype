> 기준: news-capture 캡처 화면 전체 분석 (국내 배포 / 해외 배포 / 결제 후 공통 프로세스)
> 작성일: 2026-03-30

## 0. 핵심 전제: 아임웹으로 "전부" 구현 가능한가?

### 아임웹이 기본으로 지원하는 것
- 회원가입 / 로그인 / 마이페이지
- 상품 등록 / 장바구니 / 결제 (PG 연동)
- 커스텀 HTML·CSS·JS 위젯 삽입
- 웹훅 (결제 완료 시 외부 서버 호출)
- 폼 빌더 (기본 문의 폼)

### 아임웹만으로는 불가능한 것 (커스텀 개발 필요)
| 기능 | 이유 |
|------|------|
| 매체사 다중 선택 + 실시간 가격 합산 | 동적 UI, 상태 관리 필요 |
| 단계별 주문 플로우 (3~4단계 wizard) | 아임웹 기본 폼으론 분기 처리 불가 |
| AI 원고 생성 (GPT/Claude API) | 외부 API 호출 + 결과 렌더링 |
| 해외 배포 AI Review 점수/수정 제안 | 복잡한 UI + API 응답 처리 |
| 원고 수정 확정 플로우 | 주문 상태 연동 필요 |
| 배포 리포트 | 동적 데이터 시각화 |

### 결론: 하이브리드 구조 권장

```
아임웹                          별도 웹앱 (커스텀 개발)
─────────────────────           ─────────────────────────────
홈 / 서비스 소개                  주문 플로우 전체
언론사 목록 (정적)                  (매체사 선택 → 원고 작성 → 결제)
블로그 / 고객사례                  AI 원고 생성
가격 페이지                       AI Review
회원가입 / 로그인               →  대시보드 (주문내역, 리포트)
결제 (아임웹 쇼핑몰)               원고 확정 플로우
약관 / 개인정보처리방침
```

### 2-1-A. 결제 후 원고 확정 플로우 (공통)

```

  ├── AI 원고 모드:
  │     - 입금 확인 직후 AI 자동 생성 (로딩 스피너 → 완료 시 표시)
  │     - 생성된 원고: 제목 + 본문 표시
  │
  └── 직접 작성 모드:
        - 제출한 원고 표시 (제목 + 본문)

  공통 기능:
  - [수정하기] 토글 → 제목/본문 직접 편집 → [수정 완료·저장]

  ↓ 원고 확정 완료
```

## 3. 아임웹 + 커스텀 개발 연동 설계

### 3-1. 기존 아임웹 회원과 신규 시스템 연동 방법

#### 방법 A: 아임웹을 계속 메인으로 쓰는 경우 (권장 — 기존 회원 유지)

```
아임웹 회원 DB ← 기준점

아임웹 로그인 완료
  ↓
아임웹 → 커스텀 앱으로 이동 시
  → URL 파라미터에 임시 토큰 전달
  → 커스텀 앱에서 아임웹 API로 회원 정보 조회
  → 커스텀 앱 세션 생성 (JWT)

아임웹 결제 완료 시
  → 아임웹 웹훅 → 커스텀 서버
  → 주문 상태 업데이트 + AI 원고 생성 트리거
```

**구체적 구현 순서:**
1. 아임웹 회원 가입 → 아임웹 DB에 저장
2. 주문 플로우 진입 시 아임웹 로그인 체크 → 비로그인이면 로그인 페이지로
3. 로그인 완료 후 커스텀 앱으로 리다이렉트 (토큰 포함)
4. 커스텀 앱에서 아임웹 회원 ID 기준으로 자체 회원 레코드 생성 (첫 방문 시)
5. 이후 모든 주문/원고/리포트는 커스텀 서버에서 관리
6. 결제는 아임웹 쇼핑몰 결제 or 커스텀 앱 내 PG 연동 선택

#### 방법 B: 아임웹은 랜딩만, 신규 플랫폼으로 완전 분리 (장기적으로 권장)

```
아임웹                            신규 플랫폼 (별도 도메인)
─────────────────────             ──────────────────────────
홈 / 소개 / 블로그                  회원가입 / 로그인 (새로 가입)
  ↓ "서비스 시작하기" 버튼           주문 플로우 전체
  → 신규 플랫폼 링크                대시보드
```

**기존 회원 전환 방법:**
- 아임웹 회원에게 이메일 발송: "새 플랫폼으로 이전 안내"
- 이메일로 신규 플랫폼 회원가입 유도 (기존 이메일 사용)
- 기존 아임웹 구매 이력은 마이그레이션 or 별도 안내

#### 방법 C: 아임웹 HTML 위젯에 전체 커스텀 앱 임베드 (현실적 타협안)

```
아임웹 페이지
  └── HTML 위젯 삽입
        └── <iframe src="https://app.서비스명.com/order">
              또는
            <div id="root"> (React 앱 번들 직접 삽입)
```

**장점:** 아임웹 URL 유지, 회원 세션 유지 비교적 쉬움
**단점:** iframe 보안 이슈, 모바일 대응 복잡

### 3-2. 현재 아임웹 쇼핑몰 구조 → 새 플로우 매핑

| 현재 아임웹 구조 | 새 플로우로 변경 |
|----------------|----------------|
| 상품 = "블로그 배포" 1개 | 상품 = 매체사별 개별 상품 등록 (163개+) or 패키지 상품 |
| 장바구니에 상품 추가 | 매체사 다중 선택 = 장바구니에 여러 상품 추가로 매핑 가능 |
| 결제 페이지 | 아임웹 결제 or 커스텀 결제 페이지 |
| 주문 완료 | 아임웹 주문완료 페이지 → 커스텀 완료 페이지로 리다이렉트 |

**아임웹 상품 구성 전략:**
- 국내: 매체사별 1개씩 상품 등록 → 다중 구매 → 장바구니 합산
- 해외: 패키지별 상품 3~5개 (STANDARD / BEST / PREMIUM × 국가)
- 단, 원고 작성 정보는 아임웹 주문 메모 or 커스텀 DB에 따로 저장

## 6. 아임웹 기존 회원 연동 상세 방법

### 시나리오: 아임웹에 기존 회원이 있고 새 플랫폼을 추가 구축

#### Step 1 — 아임웹에서 토큰 생성 후 커스텀 앱으로 전달

```javascript
// 아임웹 HTML 위젯에 삽입
// 아임웹 로그인 상태 확인 → 서버에 토큰 요청 → 커스텀 앱 URL로 이동

document.querySelector('#start-order-btn').addEventListener('click', function() {
  // 아임웹 회원 정보 (아임웹 JS API)
  const memberInfo = IMWEB.Member.getMemberInfo(); // 아임웹 제공 API

  if (!memberInfo.isLogin) {
    IMWEB.Member.loginPopup();
    return;
  }

  // 서버에 임시 토큰 요청
  fetch('/api/auth/imweb-token', {
    method: 'POST',
    body: JSON.stringify({ imwebMemberId: memberInfo.memberId })
  })
  .then(res => res.json())
  .then(data => {
    // 커스텀 앱으로 리다이렉트
    window.location.href = `https://app.서비스명.com/order?token=${data.token}`;
  });
});
```

#### Step 2 — 커스텀 서버에서 토큰 검증 후 자체 세션 생성

```
아임웹 회원 ID → 커스텀 DB에서 해당 ID 조회
  - 있으면: 기존 회원으로 로그인
  - 없으면: 자동 회원 생성 (아임웹 회원 정보 기반)
```

#### Step 3 — 결제 흐름 (아임웹 결제 사용으로 확정)

```
커스텀 앱에서 주문 정보 임시 저장
  (매체사 목록, 원고 내용, 담당자 정보 → 커스텀 DB에 저장, 상태: "결제대기")
  ↓
아임웹 상품 결제 페이지로 리다이렉트
  (주문 ID를 아임웹 주문 메모/옵션에 포함하여 추적)
  ↓
아임웹 결제 완료 (카드 or 무통장입금)
  ↓
아임웹 웹훅 발송 → 커스텀 서버 수신
  → 아임웹 주문 ID ↔ 커스텀 주문 ID 매칭
  → 커스텀 주문 상태 업데이트: "결제대기" → "입금확인중" (무통장) or "결제완료" (카드)
  ↓
입금 확인 후 (무통장) or 즉시 (카드):
  → AI 원고 생성 트리거
```

**아임웹 상품 구성 방법 (결제 연동을 위한 상품 설계):**

| 방식 | 설명 | 장단점 |
|------|------|--------|
| 방식 1: 금액별 가상 상품 | "1원 단위 상품"을 미리 여러 개 등록해두고 해당 금액 상품으로 이동 | 구현 복잡, 상품 수 많음 |
| 방식 2: 옵션 상품 1개 | "보도자료 배포" 상품 1개, 수량 = 총 금액으로 조작 | 단순하지만 주문 내역이 부자연스러움 |
| 방식 3: 커스텀 결제 위젯 ✅ 권장 | 아임웹 HTML 위젯에서 아임웹 결제 API 직접 호출 (아임웹 결제 모듈 활용) | 가장 자연스러운 UX, 아임웹 결제 이력도 정상 기록 |

> **권장: 방식 3** — 아임웹이 제공하는 결제 모듈(JavaScript SDK)을 커스텀 앱 내에서 직접 호출하면 UX는 커스텀 앱 안에서 완결되면서 결제는 아임웹 PG를 그대로 사용 가능

## 7. 관리자(어드민) 화면 설계 — 아임웹 내부 vs 별도 페이지

### 7-1. 관리자가 처리해야 하는 작업 목록

| 작업 | 빈도 | 설명 |
|------|------|------|
| 입금 확인 → 승인 처리 | 매일 | 무통장 입금 확인 후 주문 상태 변경 |
| AI 원고 생성 확인 | 매일 | 생성 완료 여부 확인, 실패 시 재시도 |
| 원고 수정 요청 처리 | 수시 | 고객 수정 요청 → 원고 수정 후 재전달 |
| 원고 확정 확인 | 수시 | 고객이 확정한 주문 → 배포팀에 전달 |
| 배포 완료 후 리포트 등록 | 배포 후 | 배포 결과 데이터 입력 |
| 매체사 목록 관리 | 가끔 | 신규 매체사 추가, 가격 변경, 비활성화 |
| 회원 조회 | 수시 | 고객 문의 대응 시 주문 이력 확인 |
| 쿠폰/포인트 관리 | 수시 | 발급, 취소 |

### 7-2. 각 옵션 비교

#### 옵션 A: 아임웹 내부만 사용

```
아임웹 관리자 화면
  ├── 주문 관리 → 결제 확인 O
  ├── 회원 관리 → 기본 정보 O
  └── 쿠폰/포인트 → O

아임웹에서 볼 수 없는 것:
  ✗ 매체사 선택 내역
  ✗ 원고 내용 (AI 생성 원고, 직접 작성 원고)
  ✗ 원고 수정 요청 내역
  ✗ 원고 확정 여부
  ✗ 배포 진행 상태
  ✗ 배포 리포트 등록
  ✗ 매체사 DB 관리
```

**결론: 이 서비스 구조에서는 아임웹만으로 관리 불가능**
원고 확정 여부, 배포 상태, 수정 요청 — 모두 커스텀 DB에 있어서 아임웹이 알 수 없음.

#### 옵션 B: 별도 관리자 페이지 완전 구축

```
커스텀 어드민 (별도 URL: admin.서비스명.com)
  ├── 대시보드 (오늘 주문 수, 결제대기, 원고 확정 필요 건 수 등)
  ├── 주문 관리 (전체 목록, 상태별 필터, 상세 조회)
  ├── 원고 관리 (수정 요청 목록, 확정 대기 목록)
  ├── 배포 리포트 등록
  ├── 매체사 관리 (CRUD)
  └── 회원 관리
```

**장점:** 모든 운영을 한 곳에서 처리 가능, 아임웹 왔다갔다 불필요
**단점:** 개발 비용 추가 (+60~80만원), 개발 기간 추가 (+1~1.5주)

```

─────────────────                ──────────────────────────────

                                  배포 완료 후 리포트 등록
                                  매체사 DB 관리 (CRUD)
```

**운영 흐름:**
1. 아임웹에서 입금 확인 → 아임웹 웹훅이 자동으로 커스텀 서버에 알림

4. 배포 완료 후 커스텀 어드민에서 리포트 등록
5. 매체사 추가/변경은 커스텀 어드민 매체사 관리에서 처리
